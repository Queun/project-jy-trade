# 当前项目计划

本文是当前产品能力和待办范围的唯一计划入口。已经完成的分阶段重建过程保留在 Git 历史和 `docs/archive/rebuild-development-plan.md`，不再在本文重复维护。

下一轮开发优先级尚未确定。下方“待讨论工作”只记录已知问题，不代表实施顺序。

## 产品边界

- 保留客户现有 Excel 输入和做单习惯，先把容易出错的核对与审核流程标准化。
- 旺店通只使用只读接口；不直接建单、推单、出库或修改库存。
- 系统给出匹配、库存和发货建议，审核人保留最终决定权。
- 重要操作保留批次、源文件、源行、操作者、映射和导出记录。

## 业务流程

正式订货单：

```text
导入订货单
  -> 本地商品匹配
  -> 读取本地可发库存快照
  -> 生成库存初审
  -> 人工确认发货、不发货或调整数量
  -> 生成确定发货单
  -> 导出做单 Excel
```

上游已审核确定单（进入本系统后仍须审核）：

```text
导入上游已审核确定单
  -> 本地商品匹配并读取当前成功库存快照
  -> 按实际发货数量生成建议数量和建议仓库
  -> 人工确认最终数量与最终仓库
  -> 提交审核完成
  -> 导出做单 Excel
```

确定单表达的是上游“计划发多少”，不是最终库存和仓库决定。系统以确定单 `实际发货数量` 为需求上限，复用正式订单的同商品共享库存、VIP 优先、同级公平分配和单行单仓算法。确定单中的 `主仓` / `临期仓` 不参与分配，只保留原始值用于追溯。

## 当前能力

### 账号和权限

- 用户名、密码、session cookie 和退出登录。
- 固定 `admin`、`operator`、`reviewer` 三种角色。
- `operator` 负责导入、商品同步、商品/地址维护和导出；`reviewer` 负责审核决定和优先处理；`admin` 拥有全部权限。
- 关键写操作由后端校验角色，审核和维护操作写入审计日志。
- 当前腾讯云试运行环境仅允许约定 IP 段访问，数据按低敏内部业务数据处理，优先保证操作方便；管理员允许暂时使用 `admin / yjmy`，不强制复杂密码规则或建设网页账号管理模块。
- 生产模式只自动维护环境变量指定的管理员账号，不在全新数据库中创建固定密码的测试操作员或审核员。后续需要按员工姓名创建账号时，优先增加服务器命令行账号维护工具，而不是立即开发网页用户管理。

### 订单和批次

- 从浏览器上传 `.xls` / `.xlsx` 订货单并创建正式批次。
- 按固定 Sheet 和字段读取订货单，保留原文件名和 Excel 行号。
- 导入上游已审核确定单后仍生成本系统待审核批次；确认系统建议并提交审核后才允许做单。
- 展示批次历史，并允许管理员删除批次及其关联运行数据。

### 商品匹配

- 管理员可把旺店通商品与库存自动同步设置为每 1、2、6 或 24 小时；任务按 `Asia/Shanghai` 自然时间边界执行，保留手动立即同步入口，修改周期无需重启服务。
- 手动同步同时保留“立即同步”完整流程和独立“快速同步”试用入口。快速同步回看最近 24 小时普通商品和组合装变化，只刷新受影响组件库存；它必须继承一份仓库覆盖一致、SKU 完整的成功快照，任何增量歧义、响应不完整或变化量超限都会拒绝生效并保留旧快照。
- 当前匹配顺序为：已确认长期映射 -> 普通 WDT 规格精确命中 -> WDT 组合装 -> 名称候选提示。人工确认用于纠正自动条码关系，必须覆盖普通商品或组合装直匹配；禁用或删除映射后才回退到 WDT 自动匹配。组合装按自身 `suite_no` / 条码识别，并保留全部有效组件。
- 修改长期映射时按映射记录 ID 精确覆盖，不因外部条码或货品码变化而留下旧记录；从审核行定位映射时同时查询外部条码和货品码。打开修改表单本身不改变映射状态。
- 条码、外部编码或人工保存的长期映射可以形成可靠命中；名称相似只生成候选，必须人工确认。
- 长期映射支持保存、复查、禁用和删除；确定单保存映射后自动局部应用，只重算受影响外部商品及其旧、新 SKU 共享库存池，并保留无关审核结果。
- 组合装使用全部组件的 `spec_no` 查询库存，做单导出使用组合装 `suite_no` 作为做单码。
- 小样和套盒维护表支持导入预览、确认导入、来源留痕和人工查看，但尚未接入普通订单库存初审。

### 库存和审核

- 后台组合任务使用旺店通 `wms.StockSpec.search2` 的 `available_send_stock` 生成原子库存快照；订单、重算、商品搜索和候选展示只读最新成功快照，不使用账面库存，也不回退实时 API。
- 商品档案保持全局同步，库存快照只持久化同步开始时已启用的仓库类别，并单独记录覆盖范围。多仓启用时只请求一次旺店通后在本地过滤；只启用一个已知具体仓库时可直接使用 `warehouse_no` 缩小查询范围。
- 新启用的仓库若尚未被当前快照覆盖，界面会提示具体缺失范围，相关库存建议标记为未验证，不会把未同步仓库误判为零库存；保存仓库设置不会自动启动耗时同步。
- 同步中的快照不参与业务查询；失败时清理不完整数据并继续使用上一份成功快照，最近保留两份成功快照。
- 已核验但无分仓明细的 SKU 视为库存 0；未被快照覆盖的 SKU 标记“库存未验证”，用户仍可人工确认。
- 快照超过当前自动同步周期会提醒刷新，但仍可用于审核；批次记录本次计算使用的快照 ID 和时间，后台新快照不会自动改变已有审核结果。
- 仓库按主仓、临期仓、次品仓和其他仓分类；管理员可配置哪些仓库参与建议发货。
- 普通商品和组合装共享组件库存池；组合装必须在同一仓库凑齐全部组件，每发一套同时扣减全部组件的每套用量。
- 分配固定先区分 VIP 层级，再按管理员选择的商品类型顺序处理。默认顺序为“VIP 组合装 -> VIP 普通商品 -> 普通门店组合装 -> 普通门店普通商品”；切换“普通商品优先”时只交换同一 VIP 层级内的商品类型，VIP 始终更高。
- 同一分配层级按最终商品件数逐件公平分配；不同组合装共享组件时使用同一库存池，余数按原始订单行顺序稳定处理。
- 组合装行可展开全部组件，查看每套用量、主仓/临期仓、所有分仓库存和当前建议仓的瓶颈组件；停用仓库存仍展示，但不参与系统建议。
- 一条订单明细只建议一个发货仓：可由单仓满足时优先选择满足仓，否则选择可发库存更多的仓，并列时优先主仓；不把一条明细拆到多个仓。
- 审核人可以批量通过、逐行修改发货数量和最终仓库、选择不发货、填写原因和设置优先处理。审核区支持按商品名称、外部条码、货品码、旺店通规格或做单码搜索；搜索结果唯一对应一个商品时，可把该商品在全部门店的明细一次设为不发货/不做单。批量处理保留原备注，数量归零并清空最终仓库。自动分仓只是建议，最终决定以审核结果为准。
- 确定单审核同时展示订货数量、实际发货数量、系统建议和最终发货数量。最终数量超过建议、超过原计划，或最终仓库不同于建议仓库时只提示，不阻断保存。
- 确定单把 `ambiguous` / `not_found` 单独归入高频“待映射商品”筛选，紧跟“全部”展示商品种数和受影响订单行数；真正的 `api_error` 保留在“校验异常”。
- “待映射商品”按外部货品码分组，缺货品码时回退外部条码，两者都为空时保持独立行。分组默认折叠，显示代表商品、编码、订单行数、门店数和单一映射入口；展开后查看原订单行，其他筛选仍保持逐订单明细。
- 确定单重新校验支持保留当前最终结果或按最新库存覆盖最终结果；备注和优先标记始终保留，重算后一律需要再次提交审核。
- 管理员修改共享组件优先级后不会自动改写已有批次；普通订单由运营人员显式点击“重新计算初审”后按最新设置和成功库存快照重算，并重新进入待审核。
- 库存查询失败时保留审核行和开发者错误详情。用户可以人工填写正数和仓库，但提交时必须确认本批存在未验证库存。
- 未映射商品允许人工填写正数和仓库，但提交审核时必须二次确认。确认后的做单 Excel 使用导入商品编码，缺少时回退导入条码；两者都为空仍阻止提交，保底做单不会自动创建长期映射。
- 正式初审仍要求可用商品档案；无库存快照或 SKU 未覆盖时保留审核行并标记库存未验证，不阻断人工处理。

### 地址和做单

- 门店地址支持 Excel 导入预览、确认导入和系统内维护。
- 做单地址优先按门店编码匹配，编码未命中时才按规范化门店名称匹配；历史重复记录始终使用最后更新的一条。
- 地址维护支持清空旧地址后重新导入。清空时保留 VIP 门店身份和优先分货标记，但清除其旧收件信息；普通门店旧记录直接删除。
- 做单前检查可发货行、最终仓库和缺地址门店；发货行缺少最终仓库或必要地址时阻止生成脏文件。
- 批次必须先提交审核完成才能生成任何导出；导出阶段不重新查询库存、不重新分配数量或仓库。
- 支持初审单、确定发货单和旺店通做单 Excel 三类导出。
- 导出记录持久化并提供历史和下载入口。
- 做单 Excel 只包含 `decision=ship` 且最终发货数量大于 0 的行。
- 做单 Excel 按“门店 + 最终仓库”分组生成稳定的 `原始单号`；同组所有商品行重复使用该编号，让旺店通合并为一单。不同门店或不同仓库使用不同编号。
- `客服备注` 按同一门店汇总本批次通知单号，便于仓库和客服回查。
- 普通商品的做单码通常为 `spec_no`；组合装使用 `suite_no`。具体字段契约以 `docs/excel-field-dictionary.md` 为准。

## 当前业务规则

- 不自动猜测 Excel 列名；输入文件必须符合已确认的 Sheet 和字段口径。
- 外部条码命中多个规格时进入人工确认。
- 条码不同但名称接近时只展示候选，不能自动发货。
- 小样、套装和赠品优先使用旺店通商品/组合装缓存；缓存无法表达的关系再使用人工维护数据。
- 多组件组合装的可发套数按单仓内最紧缺组件计算，不能跨仓拼套；审核人仍可超过建议数量或改仓，界面明确提示库存风险。
- 批次保存本次使用的分配策略和库存快照，审核行保存商品类型、组合装配方及各组件库存快照；后续同步或设置变化不会静默改写历史结果。
- 确定单四种数量语义固定为：`orderQty`（订货）、`plannedShipQty`（源实际发货）、`suggestedShipQty`（系统建议）、`approvedShipQty`（最终确认）。
- 确定单实际发货数量为 0 的行仍保留在审核中，默认不发货且不占库存；最终数量为 0 时不要求映射或仓库，也不进入做单文件。
- 库存、映射或仓库风险以提醒和建议为主；只有非负整数、正数行有效仓库、未映射正数行存在可输出的原始编码、保存/重算完成等形成有效做单结果所必需的条件才硬阻断。
- UI 近期保留用户熟悉的操作路径和布局，只做局部增量优化；新增界面仍需兼顾美观、清晰和实用。
- 当前低并发内部使用场景继续采用 SQLite，不优先迁移数据库。

## 待讨论工作

以下事项尚未排序，下一轮开发计划确认后再确定范围：

- 长期映射库从订单行进入时默认显示全部映射，定位当前条目时再显式筛选。
- 保存长期映射成功后立即关闭映射窗口；映射列表和候选在后台并行刷新，确定单以非模态状态展示局部应用进度，不阻塞用户继续查看其他内容。
- 支持外部维护商品的组件改绑、禁用，以及替换/赠品组件确认。
- 用更多真实样例确认外部商品编号、商品条码、套盒条码、`spec_no`、`goods_no` 和 `suite_no` 的边界。
- 为三类导出补齐字段来源、必填/可空策略和样例级回归测试，做单 Excel 优先。
- 按客户最终模板确认字段顺序、文件格式、样式、合并单元格和凭证留存要求。

## 暂缓范围

- 旺店通或仓库写接口、自动推单和自动出库。
- 完整用户管理界面、SSO 和复杂审批工作流。
- 为低并发场景提前迁移 PostgreSQL。
- 大规模 UI 重构或改变用户已经熟悉的核心工作流。
