## 2026-07-03 同步可靠性与诊断可信度

本阶段新增规则：商品档案完整性优先于流程速度。`goods.Goods.queryWithSpec` 同步失败时，不允许把已有局部缓存当作正式商品库使用。

同步策略：
- 默认 `WDT_GOODS_SYNC_PAGE_SIZE=500`，降低单页响应负载；必要时仍可通过 `--page-size` 手动覆盖，最大不超过 1000。
- 每页请求默认最多重试 3 次，间隔为 `1000,3000,8000` 毫秒。
- 失败信息必须包含接口名、时间窗口、`page_no`、`page_size`、尝试次数和原始错误。
- 如果某一页所有重试仍失败，本次同步立即停止，`wdt_goods_sync_runs.status=failed`；已 upsert 的旧缓存保留，但该 failed run 不能作为可用同步依据。

相关环境变量：
```powershell
WDT_GOODS_SYNC_START_DATE="2026-06-01"
WDT_GOODS_SYNC_PAGE_SIZE="500"
WDT_GOODS_SYNC_OVERLAP_DAYS="1"
WDT_GOODS_SYNC_MAX_RETRIES="3"
WDT_GOODS_SYNC_RETRY_DELAYS_MS="1000,3000,8000"
```

只读同步命令：
```powershell
WDT_ENV=prod npm run node:wdt:sync-goods -- full -- --start-date 2026-06-01 --page-size 500
WDT_ENV=prod npm run node:wdt:sync-goods -- incremental
```

诊断阻断规则：
- `npm run node:diagnose-order` 默认要求最近一次商品同步为 `success`。
- 如果最近一次商品同步是 `failed`、`running` 或不存在，诊断会直接停止，提示先完成商品同步。
- 仅排查时可加 `--allow-stale-cache`，报告 `summary` 会写入 `allow_stale_cache=true` 和 `cache_warning`；该报告不可作为正式审核依据。

诊断命令：
```powershell
npm run node:diagnose-order -- "ole案例文件——发货前\1订货单\订货通知单 .xls" outputs\order-match-diagnosis.xlsx
npm run node:diagnose-order -- "ole案例文件——发货前\1订货单\订货通知单 .xls" outputs\order-match-diagnosis.xlsx --allow-stale-cache
```

当前判断：官网存在接口频率/并发限制，但我们当前同步是串行请求，近期 `fetch failed` 更像瞬时连接、服务端断开或响应负载导致的问题，不像明确限流返回。无论原因如何，失败同步必须阻断正式诊断和后续审核。

## 2026-07-03 人工映射确认闭环

本阶段新增本地商品映射确认能力，用于处理“外部订货单条码/编码与旺店通条码不同，但人工确认是同一个规格”的场景。典型场景包括组合装、小样、客户侧自定义条码。

规则：
- 条码精确命中、编码精确命中、已确认人工映射命中，可以自动 `matched`。
- 名称/规格相似但条码不同，默认仍进入 `ambiguous`，不自动确认。
- 人工确认后写入 `product_mappings`，后续相同外部条码或外部编码优先命中映射。
- 未来 AI fallback 只做辅助评分和复查提示，不直接写入正式映射；正式映射仍必须由人确认。

CLI：
```powershell
npm run node:confirm-mapping -- --external-barcode 2153722460015 --external-code 5372246 --external-name "雅漾专研保湿修护面膜25ml*5片" --wdt-spec-no 3282770392869 --note "样例订单人工确认：组合装/客户侧条码映射到旺店通规格"
```

API：
- `POST /api/v1/product-mappings`：确认映射。
- `GET /api/v1/product-mappings?query=...`：查询映射。
- `PATCH /api/v1/product-mappings/:mappingId/status`：禁用映射或标记需复查。

前端：
- 审核工作台下方新增“商品映射确认”区域。
- 支持搜索本地旺店通规格缓存、填写外部条码/编码/名称、确认映射。
- 支持查询已有映射，并将映射标记为“复查”或“禁用”。
- 映射确认仍是人工动作；前端不调用任何旺店通写入接口。
- 诊断 CLI 会把 ambiguous 候选写入 `product_match_candidates`，前端“待确认候选”可直接一键带入确认表单。
- 候选接口：`GET /api/v1/product-match-candidates?query=...`。

样例订单验证结果：
- 确认前：`matched=21`，`ambiguous=19`，`not_found=0`。
- 确认映射 `2153722460015 / 5372246 -> 3282770392869` 后：`matched=40`，`ambiguous=0`，`not_found=0`，`api_error=0`。
- 新报告：`outputs/order-match-diagnosis-after-mapping.xlsx`。

- 默认要求最近一次 `wdt_goods_sync_runs.status` 为 `success`，否则阻断真实初审。
- `allowStaleCache=true` 仅用于临时排查，不作为正式审核依据。
- 商品匹配优先级仍为：人工确认映射 > 条码精确匹配 > `spec_no/goods_no/spec_code` 精确匹配 > 名称/规格候选打分 > 未匹配。
- 名称/规格候选不会自动确认；`ambiguous` 候选写入 `product_match_candidates`，供人工映射工作台处理。
- 库存查询只调用 `wms.StockSpec.search2`，按已确认的 `spec_no` 查询，不调用任何写入接口。
- 库存滚动分配按旺店通 `spec_no` 扣减，而不是按外部条码扣减，避免多个外部条码映射到同一规格时重复占用库存。
- 建议发货数当前只使用主仓 `001` 和临期仓 `LINQI`；次品仓 `CIPIN` / `defect=true` 与其他仓仅作为诊断和后续提示基础。

API 服务启动时会读取本地 `.env`，并按 `WDT_ENV` 创建只读旺店通客户端。缺少 `WDT_*` 或 `WDT_PROD_*` 凭据时，真实初审会返回明确错误，不会退回 mock 数据。

前端：
- 批次工作台新增“运行真实初审”按钮。
- 该按钮会创建 `mode=production_api` 的批次，然后调用 `run-real-review`。
- 真实初审成功后直接加载 `review_lines`，后续审核、批量通过、提交审核和 Excel 导出继续复用现有工作台。

## 2026-07-03 旺店通环境使用约定

旺店通测试 API 是虚拟测试环境，只适合验证签名、鉴权、接口字段形态和基础连通性，不能代表甲方正式店铺的商品档案、库存、仓库结构和业务状态。

因此项目后续规则固定为：
- mock 数据仅用于前端演示、单元测试和流程回归。
- 旺店通测试 API 仅用于接口契约验证，不用于真实库存判断。
- 样例订单、真实匹配、商品档案同步、库存初审等业务判断必须跑在正式 API profile 上。
- 正式 API 调用只允许只读接口，例如 `goods.Goods.queryWithSpec` 和 `wms.StockSpec.search2`。
- 在没有单独测试计划和人工确认前，禁止调用任何 `push`、`upload`、`create`、`update`、`delete` 等写入类接口。
- 如果正式只读接口返回异常、同步失败或缓存不完整，系统必须阻断正式诊断和真实初审，而不是退回测试 API 或本地 mock 数据。

本地开发默认建议 `.env` 设置：
```env
WDT_ENV="prod"
```

这不代表允许写入正式环境，只代表读取正式店铺的真实商品和库存数据。所有正式环境操作仍遵守“只读优先、失败阻断、人工确认”的规则。
